## 1. Claude Code 是什么

### 核心定义

**Claude Code 是一个会使用工具的 AI Agent（智能体）。**

### 和网页版 Claude 的本质区别

| | 网页版 Claude | Claude Code |
|---|---|---|
| **本质** | 聊天工具 | Agent（智能体） |
| **能力** | 只能"说" | 能"说"也能"做" |
| **有没有手** | 没有 | 有 |

**网页版 Claude**：被关在玻璃房子里的天才。能聊任何话题、能解释任何概念、能写代码——但**只能说，不能做**。你让他改文件，他只能把代码贴给你让你自己复制粘贴。

**Claude Code**：同一个天才，但给了他一套工具：
- **Read**（读文件）
- **Write / Edit**（写文件 / 改文件）
- **Bash**（运行命令）
- **Grep / Glob**（搜索）
- 等等

现在他不光能"想"，还能**主动伸手去拿信息、主动动手去改东西**。他有手了。

### 关键术语：Agent

**Agent（智能体）** = 会使用工具的 AI。

- 普通聊天 AI = 一个会说话的大脑
- Agent = 一个会说话**还会动手**的大脑

### 一句话记忆

> **Claude Code 是一个 Agent，可读可写可操作；网页版就是一个聊天工具。**

## 2. CLAUDE.md 文件

### 核心定义

**CLAUDE.md 是 Claude Code 的"项目说明书"**——告诉它"我是谁、我要干什么、你该怎么帮我"。启动时自动加载。

### 两个级别

| 级别 | 路径 | 作用范围 | 类比 |
|---|---|---|---|
| **用户级** | `~/.claude/CLAUDE.md` | 所有项目都生效 | 个人生活习惯 |
| **项目级** | `./CLAUDE.md`（项目根目录） | 只对当前项目生效 | 去朋友家做客的规矩 |

### 关键理解：叠加关系，不是包含关系

Claude Code 启动时会**同时读取两份 CLAUDE.md**，都装进上下文窗口，**同时生效**。

- 用户级和项目级**不是包含关系**
- 两套规则**叠加生效**
- 如果冲突，**项目级优先**（"离当下越近的规则越有话语权"）

### 生动比喻

> **用户级 CLAUDE.md** = 你的**个人生活习惯**（不吃香菜、说中文、喜欢先理解再动手）
>
> **项目级 CLAUDE.md** = 你**去某个朋友家做客的规矩**（进门脱鞋、猫怕生别摸、WiFi 密码是 xxx）
>
> 你去朋友家做客，个人习惯不会消失（还是不吃香菜），但要额外遵守这家的规矩（进门脱鞋）。两套规则**同时对你生效**。

### 判断规则放哪级的口诀

> **跟着我走还是跟着项目走？**
>
> - 跟着人走 → 用户级（例如"用中文回答"）
> - 跟着项目走 → 项目级（例如"这个项目使用 TypeScript 严格模式"）

### 文件命名细节（容易踩的坑）

- 文件夹 `.claude` → **小写**
- 文件名 `CLAUDE.md` → **全大写**
- **Mac 文件系统区分大小写**，`claude.md` 和 `CLAUDE.md` 是两个文件
- 全大写是约定俗成：类似 `README.md`、`LICENSE.md`，让重要文件一眼可见

## 3. 上下文窗口

### 核心定义

**上下文窗口 = Claude 每次能"看到"的信息总量上限。** Claude 4 系列大约是 **20 万 tokens**。

### 核心心智模型：一张固定大小的桌子

想象一张桌子，桌子大小固定 **20 万 tokens**。桌子上同时摆着：

1. **系统提示**（Anthropic 给的基础设定）
2. **工具定义**（Read、Write、Bash 等工具的说明）
3. **CLAUDE.md 的内容**（启动时就摆上去）
4. **从对话开始到现在的所有聊天记录**
5. **你当前问的新问题**

**关键**：桌子大小固定，东西只能往上加，不会自动消失。

### Claude 为什么会"变傻"

有**两个机制**同时作用：

#### 机制 1：桌子被挤爆（硬性上限）

聊得越久，聊天记录越来越多。快超过 20 万 tokens 时，**最老的对话会被"挤下桌"**——Claude 真的就看不见了。上午说的事，下午它可能真的"忘了"。

#### 机制 2：注意力被稀释（软性问题）

就算桌子还没爆，东西一多 Claude 也很难精准抓到重点。就像你桌上摆 1 件东西一眼就看见，摆 100 件东西要找半天。

> **⚠️ 常见误区**：Claude 不是"每次都回溯旧日记"才变傻的。它**没有日记本，只有桌面**——所有信息同时摆在桌上，桌子会被挤爆，注意力也会被稀释。

### 两个管理命令

| 命令 | 做什么 | 历史还在吗 | 适合场景 |
|---|---|---|---|
| **`/clear`** | 整张桌子**清空** | 完全没了 | 换了**完全不同的话题** |
| **`/compact`** | 把桌上内容**压缩成摘要** | 精华还在，细节没了 | **同一个任务继续做** |

### 口诀

> **换话题用 `/clear`，同任务接着干用 `/compact`。**

### `/compact` 的具体原理

让 Claude 把"到目前为止的所有对话"**自己总结一遍**，把啰嗦的、重复的、无关的扔掉，只留**关键决定、已有代码结构、未解决的问题**等精华，然后用这份浓缩摘要替换原来的对话。

**比喻**：原来桌上堆着 100 张便利贴，`/compact` 之后变成 1 张写满重点的 A4 纸——精华还在，空间腾出来了。

## 4. 权限模型

### 核心哲学（最深的一层）

**权限模型保护的是"你才是那个做决定的人"这件事本身——你对自己电脑、代码、生活的控制权。**

数据安全、隐私保护，都是这个根本目的的副产品。

### 一句话概括

> **人的位置是决策者，AI 的位置是执行者。我的权利是最大的。**

### 为什么需要权限模型

因为责任结构要求：**代码是你的，仓库是你的，电脑是你的**。代码出问题是你扛，仓库搞坏是你修，账号被盗是你哭。所以最终拍板的人**必须是你**。

Claude Code 再聪明，它的角色是"**执行者**"，不是"**决策者**"。这个设计哲学叫 **Human-in-the-loop**（人在回路中）——**AI 再聪明，关键决策也得人来按确认键**。

### 判断操作危险性的两个维度

| 维度 | 问题 | 例子 |
|---|---|---|
| **可逆性** | 这个操作能不能撤回？ | 改代码 = 可逆（git 能恢复）；`rm -rf ~/` = 不可逆（没有回收站）|
| **可见性** | 这个操作你能不能察觉？ | 修改文件 = 可见；读取 SSH 私钥 = 不可见（文件没改，但密钥可能已泄露）|

### 核心设计原则

> **越不可逆、越不可见的操作，越要让人来拍板。**

### 三类操作的权限分级

| 操作类型 | 权限态度 | 原因 |
|---|---|---|
| **读项目内文件** | 默认放行 | Agent 需要先读才能懂，而且可逆可见 |
| **读敏感区域**（`~/.ssh`、`~/.aws`）| **永远禁止** | 读敏感文件是不可见灾难 |
| **写文件** | 弹窗确认 | 会改动代码，但相对可逆 |
| **执行 Bash 命令** | 弹窗确认 | 可能极度不可逆（例如 `rm -rf`）|

### 看到弹窗该看什么：三步检查流程

弹窗跳出来，眼睛应该扫**三样东西**：

#### 第 1 步：看文件路径
- 它要动**哪个文件**？
- 是你期待的 `main.py`？还是你完全没让它碰的 `~/.zshrc`？
- **路径不对 → 立刻拒绝**

#### 第 2 步：看操作类型
- 是"修改"、"创建"、"删除"还是"执行命令"？
- 尤其看到"**删除**"和"**执行 Bash 命令**"的时候，**多停两秒**

#### 第 3 步：看具体内容（最关键）
- 如果是 Bash 命令，那条命令**具体是什么**？
- 我**看得懂吗**？
- 它跟我让 Claude 做的事**对得上吗**？

### 黄金原则 ⚠️

> **看不懂的命令，不点 Yes。**
>
> 先问 Claude："这条命令是什么意思？"搞懂了再决定。这不是胆小，这是**专业**。

### 两个避免的极端

- ❌ **极端 A**：每个弹窗都不看就点 Yes（觉得 AI 肯定对）
- ❌ **极端 B**：每个弹窗都纠结半天（什么都不敢点）
- ✅ **正确姿势**：**看一眼，对得上就过，对不上就停**。熟练后扫一眼 0.5 秒就够了，这 0.5 秒就是你和灾难之间的安全带。

### 权限的记忆功能

弹窗里其实有三个选项：
- **Yes**：这一次同意
- **Yes, and don't ask again for [这类命令]**：这次同意，以后这类命令也别问了
- **No**：拒绝

中间那个是"记忆开关"。让**弹窗只出现在真正需要你判断的时候**，避免麻木。

### 权限规则存储

- **用户级**：`~/.claude/settings.json`（所有项目生效）—— 例如信任 `ls`、`git status` 这种基础命令
- **项目级**：`./.claude/settings.json`（只对当前项目）—— 例如这个项目特有的 `npm run build`

### 危险开关：`--dangerously-skip-permissions`

- 作用：开启后，Claude Code 做任何事都不问你，所有弹窗消失
- **名字里的 `dangerously`（危险地）不是开玩笑，是 Anthropic 在用名字本身警告你"别乱用"**
- **零基础阶段绝对不要用**，因为：
  1. 你还在学习阶段，每次弹窗都是一次学习机会
  2. 你的电脑不是隔离环境，有真实资产
  3. 会养成坏习惯——一旦体验过"不被打扰"就很难回到"每次确认"

### 心智模型：带学徒证的实习医生

> Claude Code 像一个**带学徒证的实习医生**，你是**主治医师**。
>
> - 实习医生可以自己看病历（**读** = 默认放行）
> - 但开药、做手术前必须找主治签字（**写/执行** = 弹窗确认）
> - 有些操作他根本碰不了，比如开麻醉药（**敏感目录** = 永远禁止）
> - 主治可以授权"以后这类常规检查你自己签就行"（**记忆功能**）
> - 主治可以说"今天我不在你全权处理"（**dangerously-skip-permissions**）——但只有**确实合适**的场景下才能用

**关键：实习医生再聪明，也不能取代主治的判断。不是因为他不行，是因为责任结构不允许。**

## 5. MCP 服务器

### 核心定义

**MCP (Model Context Protocol，模型上下文协议) 是一套让 AI Agent 和外部工具对接的统一标准。**

类似 **USB-C 之于硬件**——让"一次对接，全世界通用"变成可能。

### 为什么需要 MCP？

Claude Code 自带的工具（Read/Write/Bash/Grep）数量有限，无法覆盖世界上所有的软件（Notion、GitHub、数据库、Figma……）。

解决思路有三种：

| 解法 | 问题 |
|---|---|
| **A. 让 Anthropic 全做了** | 世界上软件太多，做不完；而且每家都在更新 |
| **B. 让每个用户自己写代码** | 门槛太高；全世界重复劳动 |
| **C. 搞一个"标准插头"**（MCP）| 每个软件做一次适配器，全世界所有 AI 都能用 ✅ |

**没有 MCP 的世界就像没有 USB-C 的世界**：每个设备要一种专属充电线，重复劳动而且互不通用。

### 关键术语区分

| 术语 | 是什么 | 类比 |
|---|---|---|
| **MCP（协议）** | 一套**规矩/标准** | USB-C 这个**接口标准** |
| **MCP 服务器** | 按这套规矩做出来的**具体程序**（翻译官）| 一个**具体的外设**（U 盘、硬盘）|
| **MCP 客户端** | Claude Code 自己 | MacBook 上的 **USB-C 插孔** |

### ⚠️ 重要澄清："服务器"这个词有误导性

**"MCP 服务器" ≠ 云端的大机器。**

- 在编程世界里，"**服务器**（server）"的本质含义是"**提供服务的程序**"，不是"机器"
- **大多数 MCP 服务器就是运行在你自己电脑上的小程序**，和 Claude Code 是"邻居"
- 可以理解为 "**server 即服务，而不是服务器**"——这样理解就不会被中文翻译误导

### 核心比喻：不赚差价的中间商

> **MCP 服务器 = 一个不赚差价的中间商 / 中转站 / 翻译官**
>
> - 它自己**不做事**，只做**两次翻译**
> - 下行翻译：把 Claude Code 的请求翻译成外部软件能听懂的格式
> - 上行翻译：把外部软件的返回翻译成 Claude Code 能理解的格式

### 工作流程示例（以 GitHub MCP 为例）

假设你对 Claude Code 说："帮我看看 `yinlei-project` 有没有未关闭的 issue。"

1. **Claude Code 启动时**会问 MCP 服务器："你能提供哪些工具？" → 工具箱临时多出 `list_issues`、`create_issue` 等工具
2. **理解意图**：Claude Code 判断要用 `list_issues`
3. **调用工具**：通过 MCP 协议发请求给 GitHub MCP 服务器
4. **翻译执行**：GitHub MCP 服务器把请求翻译成 GitHub API 调用
5. **结果回流**：把 GitHub 返回的数据翻译回来，交给 Claude Code
6. **Claude Code 整理后用大白话告诉你**

### MCP 生态（部分常见服务器）

| 类别 | 代表 MCP 服务器 |
|---|---|
| **笔记文档** | Notion、Obsidian、Google Drive、Confluence |
| **代码开发** | GitHub、GitLab、Filesystem |
| **数据库** | PostgreSQL、SQLite、MongoDB |
| **沟通协作** | Slack、Linear、Gmail |
| **设计创作** | Figma、Blender |
| **浏览器网页** | Puppeteer（控制浏览器）、Fetch |
| **特殊能力** | Memory（长期记忆）、Sequential Thinking（结构化思考）|

### MCP 的意义

> **MCP 让 Claude Code 从"一个编程工具"进化成"能接管整个数字工作流的 Agent"——让它长出无数只手。**

### ⚠️ 关键原则：MCP 扩展能力，但不绕过权限

**装了 MCP 服务器不代表 Claude Code 能为所欲为。权限模型依然生效：**

- 调用 Notion MCP 的"**读取页面**" → 默认放行
- 调用 Notion MCP 的"**创建页面**" → **弹窗问你**
- 调用 GitHub MCP 的"**删除仓库**" → **弹窗问你**

**即使装了 100 个 MCP 服务器，你仍然是那个按确认键的人，控制权不会流失。**

### MCP 的成本（不是免费的）

1. **占用上下文窗口空间**——MCP 服务器的工具定义会被塞进桌子，装得多桌子就被占得多
2. **可能分散注意力**——工具太多时 Claude 有时会犹豫"该用哪个"
3. **安全风险**——第三方 MCP 服务器能访问你配的数据，要确保来源可信

### 使用原则

> **大任务复杂工作流才用 MCP，不要为用而用。只装当前工作流真的需要的。**

### 现阶段建议（零基础）

**现阶段先不装任何 MCP 服务器。** 原因：
1. 核心任务是打**编程基础**，MCP 是锦上添花不是雪中送炭
2. 对自带工具还没摸熟，先把 Read/Write/Bash 用熟对 Agent 感觉更扎实
3. 以后有真实痛点时再装——**等真的烦了"切项目切 Notion"的时候，装 MCP 会特别惊艳**

## 6. 子代理（Sub-agents）

### 核心定义

**子代理 = 主 Claude Code 派出的"小分身"。** 每个子代理是一个独立的 Claude 实例，有自己的上下文窗口，只负责一个具体任务，干完汇报结果后"消失"。

### 核心比喻：领导和员工

> **主 Claude Code 是领导，子代理是员工。**
>
> 领导不关心过程，只关注结果——员工怎么干活是员工的事，在员工自己的工位上完成，最后只把**一份简短汇报**交给领导。

### 典型场景：代码 review

假设你要让 Claude Code review 整个项目——找 bug、看代码风格、查安全问题、检查文档。

#### 没有子代理（串行方式）
1. 先通读代码找 bug（代码塞进上下文）
2. 再回头看风格（代码还在窗口里）
3. 再回头看安全（同一批代码第三次被看）
4. 再回头看文档（第四次）

**问题**：上下文窗口被挤爆、注意力分散、耗时长。

#### 有子代理（并行方式）

主 Claude Code 派出 4 个子代理**同时开工**：
- **子代理 A**：专找 bug
- **子代理 B**：专看风格
- **子代理 C**：专查安全
- **子代理 D**：专检文档

每个都有自己独立的上下文窗口（**4 张独立的桌子**）。完成后各自给主 Claude Code 一份**简短报告**。

### 子代理的三大威力

#### 威力 1：省空间（最核心）

**这是子代理最独特的能力——只有 AI 世界才有的神技。**

> **主 Claude Code 的上下文窗口里，只装子代理的最终报告，不装子代理干活的过程。**

具体画面：
- 子代理 A 读了 10 个文件、思考了半天 → 在 A 自己的桌子上发生
- 主 Claude 只看到一句："**发现 3 个 bug：xxx、xxx、xxx**"
- A 读的那 10 个文件、思考过程——主 Claude **一个字都看不到**

这意味着：**花很小的主桌子空间代价，得到深度分析的结果**。

**生存策略**：主 Claude 看不到子代理的过程**不是因为懒，是必须这样**。如果能看到所有过程，子代理的 10 个文件（几万 tokens）会全部倒进主桌子，主桌子瞬间爆炸，子代理机制白设计了。

#### 威力 2：省时间（并行）

多个子代理**同时工作**，总耗时约等于**最慢那个子代理的时间**，不是所有任务相加。时间被"折叠"了。

**陷阱**：不是所有任务都能并行。**有依赖关系**的任务必须串行（前一步结果是后一步输入）。Claude Code 会自动判断哪些能并行，你不用操心。

#### 威力 3：专业化（角色人设）

每个子代理可以有专门的人设和专长：

| 角色 | 职责 |
|---|---|
| `code-reviewer` | 专门看代码质量，懂 best practice |
| `test-writer` | 专门写测试用例 |
| `doc-writer` | 专门写 README 和注释 |
| `security-auditor` | 专门找安全隐患 |
| `debugger` | 专门排查 bug |

主 Claude Code 会**自动判断**："这个任务该派谁？"然后调用对应的子代理。

**为什么要专业化**：一个"啥都能干"的子代理在任何单一任务上都不如"专门干这一件事"的专家。**专业化 > 全能**。

### 一句话记忆

> **子代理 = 省空间 + 省时间 + 更专业**

### 子代理的配置

子代理通过 **Markdown 文件**（`.md` 文件）定义，和 CLAUDE.md 一脉相承：

| 级别 | 路径 | 作用范围 |
|---|---|---|
| **用户级** | `~/.claude/agents/` | 所有项目都能用 |
| **项目级** | `./.claude/agents/` | 只对当前项目有效 |

每个 `.md` 文件是一个子代理的"人设说明书"：
- 名字
- 职责是什么
- 可以用哪些工具
- 该怎么思考、关注什么

### 子代理的成本

1. **启动成本**：每召唤一个子代理都要生成新分身，简单任务可能得不偿失
2. **沟通成本**：汇报本身也要占主桌子空间
3. **理解偏差**：交代任务不清楚可能做跑偏

### 使用判断标准

| 维度 | 用子代理 | 不用子代理 |
|---|---|---|
| **任务复杂度** | 复杂任务（review 整个项目）| 简单任务（读一个文件）|
| **角度数量** | 多角度（同时要看多个方面）| 单角度 |
| **深度需求** | 要深挖但只要结论 | 浅层任务 |

### 口诀

> **大任务、多角度、要深挖 → 派子代理。**
> **小任务、单角度、轻量活 → 主 Claude 自己来。**

### 现阶段建议

**现阶段不需要自己配置子代理。** Claude Code 内置了默认子代理会自动派遣。看到输出里出现 "Launching agent..." / "Task tool" / "dispatching to sub-agent" 这种词，就知道**"哦，它派小分身了"**。

## 7. Hooks 自动化

### 核心定义

**Hooks 是"在特定事件发生时自动执行某个命令"的机制。**

### 核心句式

> **当 X 发生时，自动做 Y。**

就这么简单。

### 名字由来

想象 Claude Code 的工作流程是一条流水线：

```
你提问 → Claude思考 → [🪝钩子1] → 调用工具 → [🪝钩子2] → 工具执行 → [🪝钩子3] → 返回结果 → ...
```

**Hooks 就是挂在这些缝隙上的小挂钩**——事件一发生，挂钩上的命令就被触发。"Hook" = 钩子。

### 核心灵魂比喻：小狗迎接 🐕

> **"当我回家时（触发事件），我的小狗会开心地迎接我（自动执行的动作）。"**
>
> - 回家是你**正常会做的事**（不是为了触发小狗才回家）
> - 小狗迎接是"**顺手的温暖**"
> - 你**什么都没做**，它就发生了
> - 这是你和小狗之间**事先约定好的默契**

**Hooks 的灵魂不是"限制"，是"顺手"。** 它不让 Claude 停下来等什么，而是让 Claude **在做正事的同时，顺手帮你做点别的小事**。

### ⚠️ 常见误区

**Hooks 不是"冷却时间"、"限制"或"门槛"。**

- ❌ 错误画面：Hooks 让 Claude 做完一件事必须等一段时间才能继续
- ✅ 正确画面：Hooks 让 Claude 做事时**顺手触发一些额外动作**

区别：**冷却是"暂停"，Hooks 是"加料"。**

### Hooks 的核心三要素

每个 Hook 都是**三件事**拼起来的：

#### 1. 触发时机（什么时候执行）

Claude Code 定义了几个关键时机：

| 时机 | 说明 |
|---|---|
| `PreToolUse` | 调用工具**之前** |
| `PostToolUse` | 调用工具**之后** |
| `UserPromptSubmit` | 你提交问题之后，Claude 开始思考之前 |
| `Stop` | Claude 完成一次回复之后 |

#### 2. 触发条件（针对哪个工具/事件）

例如：
- 在 **Edit 工具**被调用**之后** = "改完文件之后"
- 在 **Bash 工具**被调用**之前** = "运行命令之前"

#### 3. 要执行的命令（自动做什么）

例如：
- `black .`（跑格式化）
- `git status`（看 Git 状态）
- `echo "Claude 改了个文件" >> log.txt`（写日志）

#### 完整例子

> **当（Edit 工具）被调用（之后），自动执行（`black .`）。**
>
> 翻译：**每次 Claude 改完代码，自动帮我跑一下格式化。**

### 4 个常见使用场景

#### 场景 1：代码格式化（最常见）
> **当 Claude 修改/创建 Python 文件之后，自动跑 `black` 或 `ruff` 格式化。**

代码永远保持整齐，不用手动管。

#### 场景 2：自动跑测试
> **当 Claude 修改代码文件之后，自动跑相关测试。**

改完立刻质检，出问题第一时间发现。相当于给 Claude Code 装了**自动质检员**。

#### 场景 3：操作日志
> **当 Claude 调用 Bash 工具之前，自动把要执行的命令写入日志。**

所有操作都可追溯，出问题能回头查。对学习阶段特别有用——可以事后复盘 Claude 的每一步操作。

#### 场景 4：敏感操作拦截
> **当 Claude 准备运行危险命令（如 `rm -rf`）时，自动拦截或发警报。**

在权限模型之外**再加一层硬性安全网**。就算某次脑子不清醒点了 Yes，这个 Hook 也会挡住。

### 四个场景的共同点

- **主要任务**：Claude 本来就要做的事（改代码、跑命令）
- **Hook 动作**："顺手"的额外事（格式化、测试、日志、拦截）

**Hooks 不是让 Claude 做主要任务，而是在主要任务发生时挂"副作用"。**

### Hooks 与其他机制的对比

| | 权限模型 | Hooks |
|---|---|---|
| **触发** | 工具被调用前 | 事件发生时 |
| **目的** | 让你**人工确认** | 让系统**自动执行** |
| **需要介入** | 需要点 Yes/No | 不需要，自动跑 |

**关键区别**：权限模型是"**问你**"，Hooks 是"**替你**"。

| | MCP | Hooks |
|---|---|---|
| **本质** | 给 Claude Code**新能力** | 在现有流程上加**自动化** |
| **比喻** | 给手机装**新 APP** | 给手机设**定时提醒** |

| | 子代理 | Hooks |
|---|---|---|
| **主角** | 做**主要任务** | 主要任务的**副作用** |
| **谁决定** | Claude 自己判断 | 你**预先定义**的规则 |

### 现阶段建议

**现阶段不用配 Hooks。** 但理由和 MCP、子代理不太一样——

**Hooks 不是提前学的，是烦出来的。**

- 自动化的前提是"已经做了很多次，烦了"
- 零基础阶段做的每件事都是新鲜的，没有"烦得要死的重复动作"
- 等某天发现"每次改完代码都要跑一遍测试，烦死了"——**那个"烦"的瞬间，就是该配 Hook 的时机**

### 记住这一句就够了

> **如果我将来发现自己在做什么重复的动作，我可以回头找 Hooks。**

Hooks 会在那里等你，等你真的需要它的那一天。

学完这 7 题后，我对 Claude Code 的理解从"**一个会写代码的 AI 工具**"升级到了"**一个有记忆管理、有权限边界、能扩展能力、能派分身、能自动化的完整 Agent 系统**"。

### 七题之间的关系网

```
          【Claude Code = Agent + 工具调用】
                      ↓
       ┌──────────────┼──────────────┐
       ↓              ↓              ↓
  【怎么配】      【怎么记】       【怎么控】
  CLAUDE.md     上下文窗口      权限模型
  （说明书）    （固定桌子）    （决策者位置）
                      ↓
       ┌──────────────┼──────────────┐
       ↓              ↓              ↓
  【怎么扩】      【怎么拆】       【怎么自动】
   MCP           子代理           Hooks
  （扩能力）    （拆任务）       （加顺手）
```

### 贯穿始终的原则

1. **你是决策者，AI 是执行者**（第 4 题的核心，贯穿所有题）
2. **上下文窗口是稀缺资源**（第 3 题的核心，影响 MCP 和子代理的使用判断）
3. **能力扩展不绕过权限**（MCP 和 Hooks 都遵守权限模型）
4. **痛点在前，工具在后**（现阶段不急着用 MCP/子代理/Hooks，等真有痛点再学）

### 学习方法论收获

除了知识本身，这次学习过程还让我学到了一些**怎么学东西**的方法：

1. **能用自己的话讲出来才是真的懂**——背定义没用，用自己的语言重新铸造一遍才是"学会"
2. **造比喻 → 检查比喻 → 修正比喻**——从"冷却时间"到"小狗迎接"的切换证明了这个方法的威力
3. **识别"松鼠冲动"**——学新东西时想"再多抓一点"的念头要警惕，没到时候的知识会蒸发
4. **学习冲动 ≠ 现在就该学**——知识有保鲜期，痛点是最好的老师
